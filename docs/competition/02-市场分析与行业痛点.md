# 2 市场分析与行业痛点分析

## 2.1 需求分析

<!-- 图——需求分析（使用 diagrams/requirement-analysis.drawio） -->

建筑能源智能管理系统的需求来源于建筑运维的实际业务场景，主要面向建筑运维管理人员、能源管理部门和设备维护团队。通过深入调研建筑能源管理业务流程，结合赛题用户期望，本项目从以下维度梳理核心需求：

### 2.1.1 能耗数据统一管理与高效查询的需求

建筑运行过程中产生的能耗数据涵盖电力、水、燃气、暖通空调等多种类型，数据来源于分散的计量设备和传感器，存在格式不一、标准不同的问题。运维人员在进行能耗分析时，往往需要跨系统、跨平台调取数据，手动整理格式，耗费大量时间。

以办公类建筑为例，一栋10层办公楼通常配备电力总表1个、楼层分表10个、暖通空调系统温度传感器20余个、流量计4-6个，每小时可产生数十条能耗记录。面对如此大量的时序数据，传统的人工查表、Excel统计方式在效率和准确性上都难以满足快速决策的需求。

因此，系统需要：
- 提供标准化的数据导入接口，支持CSV、JSON等通用格式
- 构建统一的能耗数据模型，涵盖建筑、设备、时间等核心维度
- 实现按建筑、时段、能耗类型、监测参数等多条件组合精准查询
- 支持与AI大模型的标准化数据交互，使智能助手可直接查询能耗数据

### 2.1.2 多维度统计分析与可视化的需求

建筑能源管理不仅需要原始数据的查询，更需要从数据中提炼出有价值的分析结论。运维人员日常需要完成的分析任务包括：

- **时段汇总统计**：按小时、日、周、月等粒度汇总各类能耗数据，了解能耗变化趋势
- **COP（制冷系数）计算**：暖通空调系统的COP是衡量制冷效率的核心指标，需要结合供回水温度、流量和电力消耗进行计算
- **异常数据分析**：识别能耗突增、COP异常下降、温度偏差等异常事件，及时预警
- **对比分析**：不同建筑、不同时段的能耗横向对比，发现节能空间

这些分析任务若依赖人工完成，不仅效率低下，还容易出现计算错误。系统需要至少实现3类核心统计分析、支持至少2种可视化图表展示，并可自动生成导出统计报表。

### 2.1.3 智能化运维辅助的需求

建筑运维知识高度依赖经验积累。一位资深运维工程师需要3-5年才能积累足够的故障诊断经验。当经验丰富的老员工离职或退休时，大量隐性知识随之流失，新员工面临漫长的学习曲线。

运维人员在日常工作中经常面临如下场景：
- "A栋大楼上周三下午的能耗为何异常偏高？"
- "冷水机组COP持续低于3.0可能是什么原因？"
- "如何排查暖通空调供回水温差过大的问题？"
- "该型号冷水机组的标准维护流程是什么？"

这些问题需要结合能耗数据、设备参数、运维手册等多源信息进行综合分析，传统的文档检索方式难以快速定位答案。因此，系统需要构建领域知识库，通过自然语言交互精准响应运维问题。

### 2.1.4 标准化数据交互协议的需求

随着AI大模型在各行业的深入应用，建筑能源管理系统需要与智能助手、分析工具等外部AI应用实现无缝对接。传统的私有API接口存在对接成本高、适配工作量大的问题。采用MCP等标准化协议，可以让Claude、GPT等主流AI助手直接通过自然语言调用能耗查询、统计分析等功能，大幅降低集成门槛。

## 2.2 行业痛点分析

<!-- 图——行业痛点分析（使用 diagrams/pain-points.drawio 或新建 diagrams/pain-points-analysis.drawio） -->

通过对建筑能源管理行业的深入调研，我们总结出以下核心痛点：

### 2.2.1 数据层面：能耗数据分散、格式不统一

建筑能源系统通常由多个子系统组成（电力系统、暖通空调系统、给排水系统、照明系统等），各子系统的数据采集设备品牌不同、协议各异、数据格式参差不齐。

**具体表现**：
- 不同品牌的智能电表输出的数据格式不一致（CSV、XML、私有协议等）
- 暖通空调系统的BACnet数据与电力系统的Modbus数据缺乏统一管理
- 历史数据存储在各自独立的数据库中，形成数据孤岛
- 人工录入数据存在滞后和错误风险

**影响**：
- 跨系统数据整合耗时长，一次完整的月度能耗报表编制通常需要2-3个工作日
- 数据质量参差不齐，影响后续分析的准确性
- 难以满足快速决策的时效性要求

### 2.2.2 运维层面：依赖人工经验、知识传承困难

建筑运维领域高度依赖经验型知识。设备故障诊断、能耗异常分析等核心运维任务，往往需要具备多年实践经验的工程师才能完成。

**具体表现**：
- 故障诊断依赖老师傅的经验判断，缺乏标准化的诊断流程
- 运维知识分散在各类手册、规程、操作指南中，查找困难
- 同一问题不同工程师可能给出不同的诊断结论
- 新员工需要3-5年才能具备独立运维能力

**影响**：
- 故障响应时间长，从发现异常到定位原因平均需要数小时
- 人员离职导致知识断层，影响运维质量的连续性
- 运维成本居高不下，人工成本占建筑运维总成本的40%以上

### 2.2.3 分析层面：缺乏智能化分析手段

当前大多数建筑能源管理系统仍以基础的数据展示和简单的阈值报警为主，缺乏深度智能化分析能力。

**具体表现**：
- 异常检测主要依赖固定阈值，无法适应不同季节、不同用能模式下的动态变化
- 缺乏多传感器联动分析能力，难以发现系统级异常
- 能耗预测精度有限，无法为节能调度提供可靠依据
- 统计报表需要人工编制，效率低且格式不统一

**影响**：
- 异常发现滞后，可能造成能源浪费和设备损坏
- 节能优化缺乏数据支撑，难以量化节能效果
- 管理决策缺乏科学依据

### 2.2.4 交互层面：系统间数据孤岛严重

建筑能源管理涉及多个系统平台，包括楼宇自控系统（BAS）、能源管理系统（EMS）、设备管理系统（FMS）等，这些系统之间缺乏统一的数据交互标准。

**具体表现**：
- 各系统采用不同的API设计规范，互操作性差
- 与外部AI工具（如智能助手）的集成需要大量定制开发
- 缺乏统一的数据查询语义，自然语言查询难以实现

**影响**：
- 系统集成成本高，项目实施周期长
- AI能力难以渗透到日常运维流程中
- 数据价值未能充分挖掘

2. **知识图谱增强**：基于LightRAG的图增强检索，突破传统RAG在跨文档信息整合上的瓶颈
3. **学术创新驱动**：融合前沿时间序列异常检测研究成果，检测能力显著优于传统阈值方案
4. **开源可扩展**：采用开源技术栈，企业可根据实际需求灵活定制

## 2.4 市场应用价值分析

### 2.4.1 经济价值

- **降本增效**：智能问答系统可替代30-50%的一线运维咨询工作，减少人力投入
- **节能减排**：通过智能化异常检测和分析，可发现5-15%的隐性能耗浪费
- **报表自动化**：自动生成统计报表，节省运维人员每周数小时的报表编制时间

### 2.4.2 技术价值

- **MCP协议示范**：为建筑行业AI化提供标准化接入的技术示范
- **RAG技术落地**：验证LightRAG在建筑运维领域的应用可行性
- **异常检测创新**：将前沿时间序列异常检测研究成果应用于实际建筑场景

### 2.4.3 社会价值

- **助力"双碳"目标**：智能化能源管理有助于降低建筑运行碳排放
- **促进技术普惠**：开源方案降低中小企业的技术门槛
- **推动行业标准化**：MCP协议的应用推动建筑能源数据交互标准化进程

### 2.4.4 市场潜力

根据市场研究数据：
- 2025年中国智慧建筑市场规模预计突破万亿元
- 建筑能源管理细分市场年均增长率超过15%
- "AI+建筑"应用场景在政策推动下加速落地
