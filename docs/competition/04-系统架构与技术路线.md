# 4 系统架构与技术路线

## 4.1 系统整体架构

AERO系统采用前后端分离的分层架构设计,整体分为数据层、服务层、接口层和表现层四个层次。数据层负责能耗数据的持久化存储与知识图谱的构建维护,采用PostgreSQL关系型数据库存储结构化能耗数据,采用LightRAG框架构建与管理领域知识图谱,采用文件系统存储导出的报表文档。服务层封装核心业务逻辑,包括数据管理服务(负责数据导入、清洗、验证)、查询统计服务(负责多条件查询、时序聚合、COP计算)、异常检测服务(负责阈值检测与SAR-AD深度检测)、智慧运维服务(负责知识图谱检索与问答生成)、报表导出服务(负责CSV/Excel报表生成)。接口层对外暴露两类接口,RESTful API基于FastAPI框架实现,提供标准的HTTP接口供前端调用,MCP Server基于FastMCP框架实现,提供标准化的MCP工具接口供AI客户端调用。表现层采用Vue 3框架构建响应式Web界面,集成Naive UI组件库与ECharts可视化引擎,提供Dashboard总览、能耗查询、统计分析、异常检测、智能问答等功能模块。

系统技术选型遵循高性能、易扩展、生态成熟的原则。后端框架选用FastAPI 0.115+,其基于Starlette与Pydantic构建,原生支持async/await异步编程模型,自动生成OpenAPI文档,性能优于传统的Flask与Django框架。数据库选用PostgreSQL 14+,其作为成熟的关系型数据库,支持复杂查询、事务处理、索引优化,通过合理的索引设计(如在能耗记录表上建立building_id与timestamp的联合索引)可实现毫秒级的时序数据查询。ORM层选用SQLAlchemy 2.0+,其提供了强大的异步ORM能力,支持声明式模型定义与复杂查询构建。知识引擎选用LightRAG 1.4.9+,其实现了图增强RAG的双层检索机制,支持增量知识更新。MCP框架选用FastMCP 2.0+,这是Anthropic官方提供的Python MCP SDK,支持Streamable HTTP传输协议。前端框架选用Vue 3.4+,其Composition API提供了更灵活的逻辑复用能力,性能优于Vue 2。UI组件库选用Naive UI 2.40+,这是一个中文友好的企业级组件库,提供了丰富的表单、表格、对话框等组件。图表库选用ECharts 5.5+,其提供了折线图、饼图、散点图等丰富的可视化图表类型。

## 4.2 数据层设计

### 4.2.1 数据库模型设计

系统数据库包含11张核心数据表,涵盖建筑信息、气象数据、能耗计量、设备台账、HVAC设备运行数据、异常事件等实体。buildings表存储建筑基本信息,包括building_id(主键)、name(建筑名称)、building_type(建筑类型,枚举值包括office/commercial/residential/hospital/school)、area(建筑面积)、address(地址)、floors(楼层数)、year_built(建成年份)、climate_zone(气候区)、design_cooling_load(设计冷负荷)、design_heating_load(设计热负荷)等字段。energy_meters表存储能耗计量数据,包括id(主键)、building_id(外键关联buildings表)、timestamp(数据采集时间戳)、total_electricity_kwh(总电力消耗)、hvac_electricity_kwh(暖通空调电力消耗)、lighting_kwh(照明电力消耗)、plug_load_kwh(插座负荷电力消耗)、peak_demand_kw(峰值需求功率)、gas_m3(燃气消耗)、water_m3(用水量)、cooling_kwh(制冷量)、heating_kwh(制热量)等字段。

equipment表存储设备台账信息,包括id(主键)、building_id(外键)、equipment_type(设备类型,枚举值包括chiller/ahu/boiler/vav/pump/cooling_tower)、name(设备名称)、model(型号)、manufacturer(制造商)、install_date(安装日期)、rated_power_kw(额定功率)、rated_capacity(额定容量)、rated_cop(额定COP)、system_type(所属系统类型)等字段。chiller_records表存储冷水机组运行数据,包括id(主键)、equipment_id(外键关联equipment表)、timestamp(时间戳)、chw_supply_temp(冷冻水供水温度)、chw_return_temp(冷冻水回水温度)、chw_flow_rate(冷冻水流量)、cw_supply_temp(冷却水供水温度)、cw_return_temp(冷却水回水温度)、cw_flow_rate(冷却水流量)、power_kw(功率)、cop(制冷系数)、load_rate(负荷率)等字段。类似地,ahu_records、boiler_records、vav_records、pump_records、cooling_tower_records表分别存储AHU、锅炉、VAV、水泵、冷却塔的运行数据。

anomaly_events表存储异常检测结果,包括id(主键)、building_id(外键)、equipment_id(外键,可为空)、timestamp(异常发生时间)、anomaly_type(异常类型,如energy_spike/low_cop/temp_deviation)、severity(严重级别,枚举值包括low/medium/high/critical)、metric_name(异常指标名称)、metric_value(异常指标值)、threshold_value(阈值或期望值)、description(异常描述)、detection_method(检测方法,如threshold/ml_model)、resolved(是否已解决)、resolution_notes(解决方案备注)等字段。

关键索引设计对查询性能至关重要。在energy_meters表上建立(building_id, timestamp DESC)联合索引,支持按建筑和时间范围的高效查询,查询计划可直接利用索引避免全表扫描。在anomaly_events表上建立(severity, resolved)联合索引,支持快速筛选未解决的高优先级异常。在各HVAC设备运行数据表上建立(equipment_id, timestamp DESC)联合索引,支持按设备和时间范围的高效查询。

### 4.2.2 数据导入与清洗流程

数据导入流程包括格式验证、去重处理、异常值过滤、时间对齐四个步骤。格式验证阶段通过Pydantic Schema定义数据模型,自动验证字段类型、必填项、取值范围等约束,例如能耗值必须为非负浮点数,时间戳必须为有效的ISO 8601格式,建筑类型必须为预定义的枚举值之一。去重处理阶段基于(building_id, timestamp)联合键进行去重,支持skip策略(跳过重复记录)与update策略(更新已有记录),通过数据库的ON CONFLICT子句实现高效的upsert操作。异常值过滤阶段过滤物理上不合理的数值,例如负能耗值、超出合理范围的温度值(如供水温度低于-50℃或高于100℃)、COP值超出物理极限(如COP大于20)等,这些异常值通常由传感器故障或数据传输错误导致,需要在入库前过滤以保证数据质量。时间对齐阶段将不同频率的数据统一对齐到标准时间间隔,例如将分钟级数据聚合为小时级数据,通过时间窗口聚合函数(如AVG、SUM)实现。

## 4.3 查询统计模块技术路线

### 4.3.1 MCP协议实现细节

MCP服务端基于FastMCP框架实现,通过装饰器模式将Python函数封装为MCP工具。以query_energy_data工具为例,其函数签名为query_energy_data(building_id: str, start_date: str, end_date: str, metrics: List[str] = None, limit: int = 100),通过@mcp.tool()装饰器注册为MCP工具,FastMCP框架自动生成工具的JSON Schema描述,包括参数名称、类型、默认值、描述等元数据。当AI客户端调用该工具时,发送JSON-RPC格式的请求{"jsonrpc": "2.0", "method": "tools/call", "params": {"name": "query_energy_data", "arguments": {"building_id": "BLD-001", "start_date": "2025-01-01", "end_date": "2025-01-31"}}},MCP服务器解析请求参数,调用对应的Python函数,执行数据库查询,将查询结果序列化为JSON格式返回给客户端。

Streamable HTTP传输协议基于Server-Sent Events(SSE)实现,支持服务器向客户端推送流式数据。MCP服务器在/mcp/路径下暴露SSE端点,客户端通过HTTP GET请求建立长连接,服务器通过event: message格式推送JSON-RPC消息。这种设计相比传统的HTTP请求-响应模式,支持双向通信与流式输出,适合大模型生成长文本回答的场景。

### 4.3.2 多条件查询引擎实现

查询引擎基于SQLAlchemy的查询构建器实现,支持建筑维度、时间维度、指标维度、排序维度、分页维度的灵活组合。查询构建逻辑采用链式调用模式,从基础查询对象开始,根据查询条件动态添加WHERE子句、SELECT子句、ORDER BY子句、LIMIT/OFFSET子句。例如,当用户指定building_id时,添加WHERE building_id = :building_id条件;当用户指定start_date和end_date时,添加WHERE timestamp BETWEEN :start_date AND :end_date条件;当用户指定metrics列表时,仅SELECT指定的字段而非SELECT *;当用户指定排序字段与排序方向时,添加ORDER BY field ASC/DESC子句;当用户指定分页参数时,添加LIMIT :limit OFFSET :offset子句。这种动态查询构建方式既保证了查询的灵活性,又避免了SQL注入风险(所有参数通过绑定变量传递)。

### 4.3.3 COP计算算法实现

COP(Coefficient of Performance)计算基于热力学第一定律,其物理含义为制冷量与输入功率的比值。计算公式为COP = Q_cooling / W_input,其中制冷量Q_cooling通过冷冻水侧的焓差计算,Q_cooling = ṁ × c_p × ΔT,ṁ为冷冻水质量流量(kg/s),c_p为水的比热容(4.186 kJ/(kg·℃)),ΔT为供回水温差(℃)。在工程实践中,通常使用体积流量(m³/h)代替质量流量,此时公式简化为Q_cooling = flow_rate_m3h × 1.163 × |T_return - T_supply|,其中1.163为单位换算系数(1 m³/h × 1000 kg/m³ × 4.186 kJ/(kg·℃) / 3600 s/h ≈ 1.163 kW/(m³/h·℃))。输入功率W_input通过HVAC能耗除以时段小时数计算,W_input = hvac_kwh / period_hours。

系统根据COP值自动评级,COP > 4.0评为优秀(Excellent),表示系统运行高效;3.0 ≤ COP ≤ 4.0评为良好(Good),表示系统运行正常;2.0 ≤ COP < 3.0评为一般(Fair),表示需要关注优化;COP < 2.0评为较差(Poor),表示需要维护检修。这一评级标准参考了GB/T 50189《公共建筑节能设计标准》与ASHRAE标准中关于冷水机组能效的要求。

## 4.4 智慧运维模块技术路线

### 4.4.1 LightRAG知识图谱构建流程

知识图谱构建流程包括文档分片、实体关系抽取、图谱构建、向量索引四个阶段。文档分片阶段将长文档切分为固定长度的文本块(默认1200字符,重叠200字符),保证每个文本块包含完整的语义单元。实体关系抽取阶段调用大语言模型(如DeepSeek-V3或Qwen2.5)对每个文本块进行分析,提取实体(如"冷水机组"、"COP"、"制冷剂泄漏"等)及其关系(如"冷水机组"--[具有指标]-->"COP"、"制冷剂泄漏"--[导致]-->"COP下降"等),输出结构化的实体-关系三元组。图谱构建阶段将抽取的三元组转化为NetworkX图结构,节点表示实体,边表示关系,节点与边均可附加属性(如实体类型、关系类型、置信度等)。向量索引阶段使用嵌入模型(如BGE-M3)将实体名称与文本块编码为1024维向量,存储在NanoVectorDB向量数据库中,支持高效的向量相似度检索。

### 4.4.2 双层检索机制实现

低层检索(Low-level Retrieval)聚焦具体实体及其直接关系。当用户查询"冷水机组COP偏低的原因"时,系统首先通过向量检索召回与"冷水机组"、"COP"、"偏低"、"原因"等关键词语义相似的实体节点,然后在知识图谱中查找这些实体节点的1-hop或2-hop邻居节点,提取相关的实体-关系子图,最后将子图中的文本块拼接为上下文输入大模型生成回答。

高层检索(High-level Retrieval)捕获跨实体的主题聚合。系统通过Louvain社区检测算法将知识图谱划分为多个主题社区,每个社区代表一个语义相关的知识聚类(如"暖通空调系统"社区、"节能策略"社区、"故障诊断"社区等)。对每个社区生成摘要文本,描述该社区包含的主要实体与关系。当用户查询"如何整体优化建筑能耗"时,系统通过向量检索召回与查询语义相似的社区摘要,提取相关社区中的所有文本块,实现跨文档信息整合。

Hybrid模式结合低层检索与高层检索的结果,通过加权融合或重排序算法选择最相关的文本块。Naive模式直接进行向量检索,不利用图结构信息,适合简单事实查询。

## 4.5 异常检测模块技术路线

### 4.5.1 阈值检测实现

阈值检测实现了8类异常检测规则,包括能耗突增(energy_spike)、COP异常(low_cop)、温度偏差(temp_deviation)、AHU同时加热制冷(ahu_simultaneous_heating_cooling)、VAV区域过热(vav_zone_overheating)、VAV区域过冷(vav_zone_overcooling)、水泵频率异常(pump_frequency_anomaly)、冷却塔逼近度异常(cooling_tower_approach_anomaly)。

能耗突增检测通过滚动窗口计算能耗均值与标准差,当当前值超过均值加2倍标准差时判定为异常,严重级别根据偏差程度分级(偏差2-3倍标准差为medium,3-4倍为high,4倍以上为critical)。COP异常检测判定COP < 2.0为异常,严重级别根据COP值分级(1.5 ≤ COP < 2.0为medium,1.0 ≤ COP < 1.5为high,COP < 1.0为critical)。温度偏差检测采用季节自适应逻辑,当室外温度 > 22℃时判定为制冷模式,期望供水温度为7℃,当|实际供水温度 - 7℃| > 5℃时判定为异常;当室外温度 ≤ 22℃时判定为制热模式,期望供水温度为45℃,当|实际供水温度 - 45℃| > 5℃时判定为异常。

### 4.5.2 SAR-AD深度检测架构

SAR-AD深度检测基于深度学习模型实现,模型输入为多传感器时间序列窗口(窗口长度通常为100个时间步),包括electricity、hvac_kwh、supply_temp、return_temp、flow_rate、outdoor_temp等多个传感器通道。模型架构包括结构学习模块、路由传播模块、多尺度融合模块、异常评分模块四个部分。结构学习模块通过图注意力网络学习传感器间的邻接矩阵,自适应发现传感器依赖拓扑。路由传播模块将时间序列特征沿着学习到的图结构进行消息传递,实现多传感器信息融合。多尺度融合模块通过多尺度时间卷积(不同膨胀率的卷积核)捕获不同时间粒度的特征。异常评分模块通过全连接层输出每个时间步的异常分数,分数超过阈值时判定为异常。

模型训练采用无监督学习范式,通过重构误差作为异常评分,正常样本的重构误差较小,异常样本的重构误差较大。训练数据使用正常运行时段的历史数据,通过滑动窗口采样生成训练样本。模型推理时,对新的时间序列窗口计算重构误差,超过动态阈值(如训练集重构误差的95分位数)时判定为异常。
